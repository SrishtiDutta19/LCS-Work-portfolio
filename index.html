The problem

Fuel tanker vehicles across India occasionally stop reporting GPS data — a condition called NRD (Non-Reporting Device). The business needed to know: is this a technical fault, a network dead zone, or deliberate GPS tampering linked to fuel theft? No framework existed to distinguish between these three causes at scale across 25 depots and thousands of monthly trips.

The analytical logic — self-developed

The core classification framework was designed independently based on domain reasoning about how GPS devices physically work:

🔴

Suspected Pilferage

Vehicle showed Route Deviation (RD) alert — moved more than 500m off approved route — then went into NRD within 48 hours. The corridor from RD point to NRD point is the suspected theft zone.

RD before NRD → dark corridor
🟣

GPS Tampering

Power Disconnect (DC) alert fired during the NRD window. Key insight: if GPS comes back online far from the depot (beyond 50% of route distance), it was not repaired at depot — device was intentionally switched off and on.

DC + NRD + far reactivation
🔵

Network Dead Zone

No DC alert, no RD alert. Tracking Interruption (TI) alert present. Device next came back online near the depot. Vehicle was on correct route — signal simply lost in a known coverage gap.

TI alert + depot reactivation

Key analytical insight — original domain reasoning

A broken GPS device cannot emit any alert until it is physically repaired. Repair happens at the depot or an authorised workshop. Therefore: if the next alert after an NRD appears far from the depot (beyond 50% of the route distance), the device was not repaired — it was deliberately turned off and back on. This single insight is the cornerstone that separates genuine device faults from intentional tampering, and makes the framework analytically defensible.

What was built

Data pipeline — Python + Pandas (vectorized)

Combined JHS (Journey History) and AOT (Alert On Trip) data — 212,000+ alert records — into a unified alert stream. Used searchsorted and merge-asof for fast next-alert lookup across 1,765 NRD events. Row-by-row iteration would have timed out; vectorized merges ran in under 30 seconds.

NRD classification engine

For each NRD event: checked a ±6-hour window for DC alerts (GPS tampering), checked 48 hours prior for RD alerts (pilferage), checked TI alerts in NRD window (network). Output: classified table with category, flags, corridor coordinates.

JRM route master — 4,363 routes extracted

Parsed 600+ JRM PDF driver card documents across 25 depots using pdfplumber to extract depot names, RO coordinates, waypoints, and known network issue locations. Cross-referenced with JRM Excel for lat/lon coordinates.

Azure Maps route API integration

Python script calls Azure Maps Route API to generate real road polylines (not straight lines) for all 4,363 routes. Output saved as GeoJSON FeatureCollection for PowerBI reference layer — gives Google Maps-style road-following route curves.

PowerBI intelligence dashboard

Multi-layer interactive map: blue JRM routes, orange RD deviation points, purple DC corridors, red NRD bubbles (sized by duration), red pilferage corridor lines (RD→NRD segment), vehicle clustering with risk-color gradient (green→orange→red→dark red), 10km impact radius zones. Cross-filtering slicers by category, depot, zone, and duration.

Map visualization — layer design

Blue — Approved JRM route (Depot → RO)

4,363 real road routes from JRM data. Loaded as GeoJSON reference layer in Azure Maps. Follows actual road network, not straight lines.

Orange — Route deviation point

Where the vehicle left its approved route (>500m off route). Start of the suspicious zone.

Purple — Power disconnect point

Where GPS power was cut. If this overlaps with NRD → GPS tampering confirmed.

Red line — Pilferage corridor (RD → NRD)

The exact road segment where vehicle was off-route AND went dark. This is the untracked stretch where product likely left the truck. Precise, short, actionable.

Red/Purple/Blue bubble — NRD alert point

Where device went silent. Colored by category. Sized by duration hours (bigger = longer NRD). Hovering shows vehicle number, duration, depot, category.

What this analysis enables

Before this framework, a non-reporting vehicle was either logged as a generic "device fault" or flagged manually with no systematic way to distinguish tampering from genuine faults. After:

Operations team

Receives exact road segment (RD→NRD coordinates) where interception is needed — not just "somewhere on this route."

Security team

Gets vehicle number, date/time, depot, and evidence chain (RD+DC+NRD) for cases that warrant FIR or driver action.

Network team

Receives list of routes and coordinates with repeated TI alerts — these are confirmed dead zones needing telecom intervention, not investigation.
